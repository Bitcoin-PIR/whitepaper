\section{Evaluation}
\label{sec:evaluation}

This section separates two kinds of evidence. First, we report measurements that are reproducible directly from the current database artifacts: logical database sizes, preprocessed table sizes, Merkle proof material, OnionPIR stores, and exact two-server DPF communication per PBC round. Second, we define the controlled benchmark matrix needed for latency and throughput claims. The distinction matters because artifact size and DPF wire cost are deterministic functions of the database layout, while query latency depends on hardware, cache state, network placement, and server concurrency.

\subsection{Current Artifact Measurements}

The current measurement snapshot uses a full Bitcoin UTXO checkpoint at height $948{,}454$ and the delta database from height $940{,}611$ to $948{,}454$. Measurements were generated by \texttt{benchmarks/collect\_artifact\_metrics.py} from the local artifact root \texttt{/Volumes/Bitcoin/data}. The script parses the cuckoo-table headers, records artifact sizes, computes DPF domain sizes, and emits both CSV files and the tables included below. The measured machine was an Apple M2 Ultra system with 24 logical cores and 64~GiB of memory; exact machine and git provenance are recorded in \texttt{benchmarks/results/environment.json}.

\input{benchmarks/results/artifact_tables}

Several immediate conclusions follow from Table~\ref{tab:artifact-sizes}. The DPF/Harmony chunk cuckoo table dominates the two-server storage footprint because each chunk bucket stores inlined chunk payloads. OnionPIR shifts this cost into the NTT-preprocessed store and the indexed FHE layout. Delta databases are much smaller than full checkpoints: for the $940{,}611 \to 948{,}454$ interval, the logical chunk payload is $324.5$~MiB compared with $3.02$~GiB for the full checkpoint, and the DPF/Harmony chunk cuckoo table is $1.10$~GiB compared with $10.49$~GiB.

Table~\ref{tab:dpf-round-costs} gives the exact two-server DPF communication for one PBC round at the INDEX and CHUNK levels. These are not extrapolated timing measurements: they are derived from the current table headers. The full checkpoint uses $n=20$ for INDEX and $n=21$ for CHUNK, while the measured delta uses $n=16$ and $n=17$. The smaller delta domain lowers DPF key size, though the bucket response size remains fixed by the slot layout.

\subsection{Pilot Local Scan Benchmark}

As a first timing measurement, we benchmark the local sequential scan component over the same cuckoo artifacts. The runner \texttt{benchmarks/benchmark\_scan\_latency.py} compiles a small Rust scanner and performs one warmup plus 30 measured warm-cache trials per artifact. The scanner skips the cuckoo header, streams the table body, and touches one byte per 64-byte cache line to prevent the read loop from being optimized away. This is not an end-to-end DPF or HarmonyPIR benchmark: it does not include DPF key generation, DPF expansion, conditional XOR of selected bins, client-side verification, WebSocket framing, network latency, or Merkle proof queries. It is useful because the two-server online path is dominated by scanning these large tables, and the measurement shows the scale of that component under a fixed local environment.

\input{benchmarks/results/scan_latency_table}

The full checkpoint's CHUNK table scan is the largest measured pass: $10.49$~GiB scanned with a median time of $1.091$~s and p95 time of $1.673$~s across 30 warm-cache trials. The corresponding full INDEX scan is $2.06$~GiB with median time $0.213$~s. The measured deltas are smaller by roughly an order of magnitude and therefore have sub-200~ms median scan times in this local microbenchmark. These numbers should be interpreted as a local table-scan component, not a deployed wallet sync latency.

\subsection{Pilot DPF Round Benchmark}

The next timing layer benchmarks a complete local DPF round over the real cuckoo artifacts. The runner \texttt{benchmarks/benchmark\_dpf\_round\_latency.py} builds a small Rust benchmark against \texttt{pir-runtime-core::eval}, memory-maps each current INDEX and CHUNK artifact, generates fresh DPF key shares for random per-group targets, evaluates both local server shares, XORs the two responses, and verifies that each recovered result equals the selected real cuckoo bin. The benchmark uses one warmup and five measured trials per artifact.

\input{benchmarks/results/dpf_round_latency_table}

Table~\ref{tab:dpf-round-benchmark} is the first protocol-path latency table over the current real artifacts. Across all six artifact/level combinations, the runner verified $4{,}650$ selected bins with zero failures. For the full checkpoint, the local two-server critical path is $0.522$~s for INDEX and $0.502$~s for CHUNK at the median, where server time is the slower of the two local server-share evaluations. The smaller deltas show the expected scale trend: the $940{,}611 \to 948{,}454$ delta has median total DPF-round times of $0.034$~s for INDEX and $0.043$~s for CHUNK, while the $940{,}611 \to 944{,}000$ delta is $0.023$~s and $0.031$~s. These measurements include DPF key generation, server-side DPF expansion and table XOR, and client-side response XOR/verification. They still exclude WebSocket framing, real client query assignment, Merkle proof retrieval, and network latency.

\subsection{OnionPIR Synthetic-Capacity Phase Benchmark}

We also record one interim OnionPIR timing run from the current runtime benchmark. The runner \texttt{benchmarks/benchmark\_onionpir\_synthetic.py} invokes \texttt{cargo run --release -p runtime --bin onionpir\_bench} in the implementation repository, parses the benchmark output, and emits both machine-readable CSV and the table below. This benchmark uses the OnionPIR random-data path at the same per-group capacities used by the runtime: $8{,}224$ requested INDEX entries and $43{,}053$ requested CHUNK entries.

\input{benchmarks/results/onionpir_synthetic_table}

Table~\ref{tab:onionpir-synthetic} gives phase-level capacity data only. In this run, the INDEX capacity used a 136~MB NTT store and averaged $0.303$~s per server answer across 10 rounds; the CHUNK capacity used a 680~MB NTT store and averaged $0.581$~s. Both capacities used a 3.00~MB client key upload, 32.01~KB query, and 11.00~KB response. The benchmark is useful for the relative scale of OnionPIR FHE phases, but it is not a final real-artifact result. The current \texttt{onionpir\_bench} binary calls \texttt{PirServer::gen\_data(\&[])}, so the generated random server data does not match the deterministic \texttt{entries} vector used by the verification printout; the raw benchmark log accordingly reports a mismatch. Therefore, these numbers are not used as correctness evidence or as end-to-end deployed wallet latency.

\subsection{Latency and Throughput Benchmark Matrix}

The remaining end-to-end evaluation should benchmark the same implementation under multiple scales and environments. A publishable latency table should not use a single full-scale run alone: it should show the trend as the database grows and separate preprocessing costs from online query costs.

For DPF-PIR, the primary scaling variable is the number of bins per group. The benchmark should report INDEX and CHUNK separately because the INDEX response is small but drives chunk discovery, while CHUNK scans dominate payload retrieval. The current exact communication table gives the wire cost, and the pilot DPF runner now measures local key generation, per-server DPF evaluation, table XOR, and client-side response verification. A final deployed benchmark must additionally cover real wallet query assignment, WebSocket framing, network placement, Merkle proof verification, and at least 30 trials per scale.

For HarmonyPIR, offline and online measurements must be separated. The offline hint phase is paid when the client prepares a stateful session and should be reported as a one-time cost per database and per PRP backend. The online phase should report request construction, server response time, response processing, and query budget consumption. The paper should include both HMR12 and FastPRP results if both are supported by the deployment being evaluated.

For OnionPIR, the evaluation must distinguish database preprocessing from online queries. The NTT store is large but amortized across all clients. Per-session client key generation and key upload should be reported separately from per-query upload, server answer time, response size, and decryption time. Because the OnionPIR benchmark currently includes a self-contained random-data path, final paper numbers should either run against the current real artifacts or explicitly label the results as synthetic-capacity measurements.

\subsection{Reproducibility Artifacts}

All benchmark-backed claims in this section should be regenerated from committed artifacts rather than copied from terminal output. The current repository records:
\begin{itemize}[nolistsep,leftmargin=*]
    \item \texttt{benchmarks/results/artifact\_metrics.csv}: logical and preprocessed artifact sizes for the measured databases.
    \item \texttt{benchmarks/results/dpf\_round\_costs.csv}: exact DPF domain and communication costs derived from cuckoo headers.
    \item \texttt{benchmarks/results/dpf\_round\_latency\_trials.csv}: per-trial local DPF keygen, server evaluation, and client verification timings over real artifacts.
    \item \texttt{benchmarks/results/dpf\_round\_latency\_summary.csv}: median and tail summaries derived from the DPF round trials.
    \item \texttt{benchmarks/results/dpf\_round\_latency\_environment.json}: command, git revision, and machine metadata for the DPF round run.
    \item \texttt{benchmarks/results/scan\_latency\_trials.csv}: per-trial local scan timings for INDEX and CHUNK cuckoo artifacts.
    \item \texttt{benchmarks/results/scan\_latency\_summary.csv}: median and tail summaries derived from the scan trials.
    \item \texttt{benchmarks/results/onionpir\_synthetic\_summary.csv}: parsed OnionPIR synthetic-capacity phase results.
    \item \texttt{benchmarks/results/onionpir\_synthetic\_stdout.txt}: raw runtime benchmark log used to generate the OnionPIR table.
    \item \texttt{benchmarks/results/onionpir\_synthetic\_environment.json}: runtime command, git revision, and machine metadata for the OnionPIR run.
    \item \texttt{benchmarks/results/environment.json}: hardware, operating system, and git revision metadata for the collection run.
    \item \texttt{benchmarks/results/artifact\_tables.tex}, \texttt{benchmarks/results/scan\_latency\_table.tex}, \texttt{benchmarks/results/dpf\_round\_latency\_table.tex}, and \texttt{benchmarks/results/onionpir\_synthetic\_table.tex}: generated tables included in this paper.
\end{itemize}

The next benchmark step is to extend the DPF runner from local warm-cache trials to deployed WebSocket trials, and to add comparable end-to-end runners for HarmonyPIR hint and online phases, OnionPIR real-artifact key/query/answer/decrypt phases, and Merkle proof verification. Those runners should preserve the same one-row-per-trial CSV discipline and add cache-state, network environment, and client/runtime labels, so the paper can report medians and tail latencies without hand-editing results.
