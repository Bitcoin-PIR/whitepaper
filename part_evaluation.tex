\section{Evaluation}
\label{sec:evaluation}

This section separates two kinds of evidence. First, we report measurements that are reproducible directly from the current database artifacts: logical database sizes, preprocessed table sizes, Merkle proof material, OnionPIR stores, and exact two-server DPF communication per PBC round. Second, we define the controlled benchmark matrix needed for latency and throughput claims. The distinction matters because artifact size and DPF wire cost are deterministic functions of the database layout, while query latency depends on hardware, cache state, network placement, and server concurrency.

\subsection{Current Artifact Measurements}

The current measurement snapshot uses a full Bitcoin UTXO checkpoint at height $948{,}454$ and the delta database from height $940{,}611$ to $948{,}454$. Measurements were generated by \texttt{benchmarks/collect\_artifact\_metrics.py} from the local artifact root \texttt{/Volumes/Bitcoin/data}. The script parses the cuckoo-table headers, records artifact sizes, computes DPF domain sizes, and emits both CSV files and the tables included below. The measured machine was an Apple M2 Ultra system with 24 logical cores and 64~GiB of memory; exact machine and git provenance are recorded in \texttt{benchmarks/results/environment.json}.

\input{benchmarks/results/artifact_tables}

Several immediate conclusions follow from Table~\ref{tab:artifact-sizes}. The DPF/Harmony chunk cuckoo table dominates the two-server storage footprint because each chunk bucket stores inlined chunk payloads. OnionPIR shifts this cost into the NTT-preprocessed store and the indexed FHE layout. Delta databases are much smaller than full checkpoints: for the $940{,}611 \to 948{,}454$ interval, the logical chunk payload is $324.5$~MiB compared with $3.02$~GiB for the full checkpoint, and the DPF/Harmony chunk cuckoo table is $1.10$~GiB compared with $10.49$~GiB.

Table~\ref{tab:dpf-round-costs} gives the exact two-server DPF communication for one PBC round at the INDEX and CHUNK levels. These are not extrapolated timing measurements: they are derived from the current table headers. The full checkpoint uses $n=20$ for INDEX and $n=21$ for CHUNK, while the measured delta uses $n=16$ and $n=17$. The smaller delta domain lowers DPF key size, though the bucket response size remains fixed by the slot layout.

\subsection{Pilot Local Scan Benchmark}

As a first timing measurement, we benchmark the local sequential scan component over the same cuckoo artifacts. The runner \texttt{benchmarks/benchmark\_scan\_latency.py} compiles a small Rust scanner and performs one warmup plus 30 measured warm-cache trials per artifact. The scanner skips the cuckoo header, streams the table body, and touches one byte per 64-byte cache line to prevent the read loop from being optimized away. This is not an end-to-end DPF or HarmonyPIR benchmark: it does not include DPF key generation, DPF expansion, conditional XOR of selected bins, client-side verification, WebSocket framing, network latency, or Merkle proof queries. It is useful because the two-server online path is dominated by scanning these large tables, and the measurement shows the scale of that component under a fixed local environment.

\input{benchmarks/results/scan_latency_table}

The full checkpoint's CHUNK table scan is the largest measured pass: $10.49$~GiB scanned with a median time of $1.091$~s and p95 time of $1.673$~s across 30 warm-cache trials. The corresponding full INDEX scan is $2.06$~GiB with median time $0.213$~s. The measured deltas are smaller by roughly an order of magnitude and therefore have sub-200~ms median scan times in this local microbenchmark. These numbers should be interpreted as a local table-scan component, not a deployed wallet sync latency.

\subsection{DPF Round Benchmark}

The next timing layer benchmarks a complete local DPF round over the real cuckoo artifacts. The runner \texttt{benchmarks/benchmark\_dpf\_round\_latency.py} builds a small Rust benchmark against \texttt{pir-runtime-core::eval}, memory-maps each current INDEX and CHUNK artifact, generates fresh DPF key shares for random per-group targets, evaluates both local server shares, XORs the two responses, and verifies that each recovered result equals the selected real cuckoo bin. The benchmark uses one warmup and 30 measured trials per artifact.

\input{benchmarks/results/dpf_round_latency_table}

Table~\ref{tab:dpf-round-benchmark} is the first 30-trial protocol-path latency table over the current real artifacts. Across all six artifact/level combinations, the runner verified $27{,}900$ selected bins with zero failures. For the full checkpoint, the local two-server critical path is $0.098$~s for INDEX and $0.220$~s for CHUNK at the median, where server time is the slower of the two local server-share evaluations. The corresponding p95 server times are $0.151$~s and $0.437$~s. The smaller deltas show the expected scale trend: the $940{,}611 \to 948{,}454$ delta has median total DPF-round times of $0.009$~s for INDEX and $0.023$~s for CHUNK, while the $940{,}611 \to 944{,}000$ delta is $0.005$~s and $0.013$~s. These measurements include DPF key generation, server-side DPF expansion and table XOR, and client-side response XOR/verification. They still exclude WebSocket framing, real client query assignment, Merkle proof retrieval, and network latency.

\subsection{Pilot HarmonyPIR Offline Hint Benchmark}

HarmonyPIR separates a heavier offline hint phase from a lighter stateful online query phase. The runner \texttt{benchmarks/benchmark\_harmony\_hint\_latency.py} benchmarks the offline phase over the current real cuckoo artifacts. It builds a small Rust benchmark against \texttt{harmonypir} and \texttt{harmonypir-wasm}, memory-maps each INDEX and CHUNK artifact, and generates one complete set of per-group hints for both the HMR12 PRP and FastPRP backends. The run uses one warmup and three measured trials per artifact/backend combination.

\input{benchmarks/results/harmony_hint_latency_table}

Table~\ref{tab:harmony-hint-benchmark} shows that FastPRP is the practical backend for offline hint refreshes at current scale. For the full checkpoint, FastPRP generates the complete INDEX hint set in $0.917$~s and the complete CHUNK hint set in $1.934$~s at the median, compared with $13.725$~s and $26.128$~s for HMR12. The full-checkpoint hint material is modest: 4.0~MiB for INDEX and 14.7~MiB for CHUNK. The measured deltas are smaller and faster, with the $940{,}611 \to 948{,}454$ delta taking $0.077$~s for INDEX and $0.190$~s for CHUNK under FastPRP. These measurements cover offline hint generation only. They do not yet include hint download, client state serialization, online request construction, query-server response, response processing, hint relocation, or network latency.

\subsection{HarmonyPIR Online Query Benchmark}

The online HarmonyPIR runner \texttt{benchmarks/benchmark\_harmony\_online\_latency.py} extends the hint benchmark to the stateful query path. For each real cuckoo artifact and PRP backend, it precomputes per-group hints, initializes one \texttt{HarmonyGroup} client per PBC group, builds one fixed-count padded request per group, serves those requests from the local memory-mapped artifact, processes responses, applies the wrapper's relocation/update logic, and verifies each recovered bin against the real table. The table separates one-time client state setup from the online request/server/process path. It uses one warmup and 30 measured trials per artifact/backend combination.

\input{benchmarks/results/harmony_online_latency_table}

Table~\ref{tab:harmony-online-benchmark} closes the main HarmonyPIR measurement gap left by the offline hint table. Across 360 measured rows, the runner verified $27{,}900$ group queries with zero failures. For the full checkpoint under HMR12, median online time is $0.601$~s for INDEX and $0.893$~s for CHUNK, with p95 online times of $0.938$~s and $1.833$~s. The same full-checkpoint path under FastPRP is $3.662$~s and $5.896$~s at the median: FastPRP remains the practical offline hint-generation backend, but the current wrapper's online random-access path is slower than HMR12 and should be treated as an implementation optimization target rather than a protocol limit. The measured deltas show the expected scaling trend; for the $940{,}611 \to 948{,}454$ delta, HMR12 online medians are $0.178$~s for INDEX and $0.280$~s for CHUNK. The online columns include request construction, query-server response over the mapped artifact, client response processing, and hint relocation. They do not include network latency, browser/WASM timing, hint download, wallet query assignment, or Merkle proof verification. The setup column is reported separately because it includes client state construction and hint loading and is not paid on every online query.

\subsection{Bucket-Merkle Proof Benchmark}

The bucket-Merkle runner \texttt{benchmarks/benchmark\_merkle\_latency.py} measures the integrity layer over the same real artifacts. Each trial chooses one randomized bin in every PBC group, computes the bin leaf hash, retrieves each stored sibling level through the same DPF sibling-table evaluator used by the server, XORs the two local server shares, walks the arity-8 proof, and then closes the proof through the cached tree-top blob. The table reports the local warm-cache parallel critical path for sibling retrieval and client verification; it excludes WebSocket framing, network latency, and the primary data-query round that produced the bin being verified.

\input{benchmarks/results/merkle_latency_table}

Table~\ref{tab:merkle-proof-benchmark} adds 30-trial proof-path coverage for all three measured scales. Across 180 measured rows, the runner verified $13{,}950$ per-group bin proofs with zero failures. For the full checkpoint, the median local proof path is $0.012$~s for INDEX and $0.021$~s for CHUNK, with p95 totals of $0.039$~s and $0.024$~s. The larger measured delta has median proof totals of $0.002$~s for INDEX and $0.004$~s for CHUNK, while the smaller delta is $0.002$~s and $0.003$~s. The full-checkpoint proof path transfers only tens of KiB per server in this local DPF sibling-retrieval model: 56.8~KiB upload and 56.2~KiB download for INDEX, and 64.3~KiB upload and 60.0~KiB download for CHUNK.

\subsection{OnionPIR Synthetic-Capacity Phase Benchmark}

We also record one interim OnionPIR timing run from the current runtime benchmark. The OnionPIR synthetic runner invokes \texttt{cargo run --release -p runtime --bin onionpir\_bench} in the implementation repository, parses the benchmark output, and emits both machine-readable CSV and the table below. This benchmark uses the OnionPIR random-data path at the same per-group capacities used by the runtime: $8{,}224$ requested INDEX entries and $43{,}053$ requested CHUNK entries.

\input{benchmarks/results/onionpir_synthetic_table}

Table~\ref{tab:onionpir-synthetic} gives phase-level capacity data only. In this run, the INDEX capacity used a 136~MB NTT store and averaged $0.303$~s per server answer across 10 rounds; the CHUNK capacity used a 680~MB NTT store and averaged $0.581$~s. Both capacities used a 3.00~MB client key upload, 32.01~KB query, and 11.00~KB response. The benchmark is useful for the relative scale of OnionPIR FHE phases, but it is not a final real-artifact result. The current \texttt{onionpir\_bench} binary calls \texttt{PirServer::gen\_data(\&[])}, so the generated random server data does not match the deterministic \texttt{entries} vector used by the verification printout; the raw benchmark log accordingly reports a mismatch. Therefore, these numbers are not used as correctness evidence or as end-to-end deployed wallet latency.

\subsection{Latency and Throughput Benchmark Matrix}

The remaining end-to-end evaluation should benchmark the same implementation under multiple scales and environments. A publishable latency table should not use a single full-scale run alone: it should show the trend as the database grows and separate preprocessing costs from online query costs. The repository records the detailed benchmark protocol, environment classes, and promotion criteria in \texttt{benchmarks/BENCHMARK\_PROTOCOL.md}.

For DPF-PIR, the primary scaling variable is the number of bins per group. The benchmark should report INDEX and CHUNK separately because the INDEX response is small but drives chunk discovery, while CHUNK scans dominate payload retrieval. The current exact communication table gives the wire cost, and the DPF runner now measures local key generation, per-server DPF evaluation, table XOR, and client-side response verification. A final deployed benchmark must additionally cover real wallet query assignment, WebSocket framing, network placement, and composition with the Merkle proof path.

For HarmonyPIR, offline and online measurements must be separated. The pilot hint benchmark reports the one-time offline cost per database and PRP backend, and the 30-trial online runner reports local request construction, query-server response time, response processing, hint relocation, and query-budget consumption over real artifacts. The remaining HarmonyPIR work is to add networked hint download, browser/WASM client timing, and deployed query-server placement. The paper should keep both HMR12 and FastPRP results while both are supported by the deployment being evaluated.

For OnionPIR, the evaluation must distinguish database preprocessing from online queries. The NTT store is large but amortized across all clients. Per-session client key generation and key upload should be reported separately from per-query upload, server answer time, response size, and decryption time. Because the OnionPIR benchmark currently includes a self-contained random-data path, final paper numbers should either run against the current real artifacts or explicitly label the results as synthetic-capacity measurements.

The next benchmark step is to extend the DPF, HarmonyPIR, and Merkle runners from local warm-cache trials to deployed WebSocket trials and replace the OnionPIR synthetic-capacity run with a real-artifact correctness-preserving run. Those runners should preserve the same one-row-per-trial CSV discipline and add cache-state, network environment, and client/runtime labels, so the paper can report medians and tail latencies without hand-editing results.
